## 事実・観測 `[FACT]`

- Atlantisの標準開発入口には、ローカル`.venv`再構築、標準Dev Container、Git追跡対象だけを
  一時展開するclean-room試験がすでにある。
- 現ホストではDocker／Podmanの実行環境を観測できていないため、標準Dev Containerを含む
  container imageの実buildは未試験である。
- PythonのVenvは依存関係の分離単位であり、OS kernel、host filesystem、network、browser processを
  隔離するsandboxではない。
- SaaS AI、検索connector、Git Web検索が返す内容は、取得時点と原典更新時点が一致するとは限らない。
  ただし、各vendor内部でどの検疫、cache、index、batch処理が使われているかは本noteでは未観測である。
- 2026-07-18に手元Safari MCPでAtlantisのGitHub README、raw README、hero、tutorial導線を観測し、
  raw READMEとlocal READMEのSHA-256一致を確認した。
- 同日にMarkdownから抽出した外部URL 21件を手元Safariで観測し、20件へ到達、1件のGitHub
  `File not found`を発見した。欠損はManifestのVenv虎の巻ファイル名変更によるもので、正しいURLへ訂正した。
- 文書追加とURL訂正後の2026-07-18T18:38:04+09:00に、抽出された外部URL 24件を同じ手元Safariの
  公開・read-only経路で再観測し、24件すべてでfinal URLとpage titleを取得した。HTTP status、安全性、
  内容正当性を検証したという意味ではない。
- 手元browserで第三者pageを実行しても、DOMや本文をMCPからcoding agentへ返した範囲はSaaS contextへ入る。
  手元browser経路だけでlocal-only処理が完成するわけではない。

- 上記はApple公式文書の役割記述を観測した証跡であり、このhostで各機能が有効、最新、十分であることは
  実測していない。Atlantisの標準構成、推奨製品、安全性保証へも昇格しない。

## 考察 `[INTERPRETATION]`

検索面の回答と、実ブラウザによる観測は別の証拠クラスとして扱う必要がある。ブラウザで開いたことも
原典の鮮度や安全性を保証しないため、「live」「最新」「安全」を一語で済ませず、次のreceiptへ分離する。

| 観測クラス | 得られるもの | 得られないもの |
|---|---|---|
| `INDEXED_SNAPSHOT` | SaaS／検索面が返したindex、cache、snapshot | 原典の現在状態、vendor内部処理 |
| `HTTP_OBSERVED` | 直接HTTP応答のstatus、headers、取得hash | JavaScript実行後DOM、origin freshness |
| `BROWSER_OBSERVED` | headless browserのfinal URL、DOM、実行結果 | origin freshness、安全性の証明 |
| `ORIGIN_REVISION_MATCH` | 指定revision／artifactと取得物の一致 | 内容そのものの正しさ、安全性 |

ここでいう「コンテナ結界」は完全防御の宣言ではない。標準の編集・対話環境からbrowser executionを
分け、壊れた時に捨てる単位を明示する工学上の隔離profileである。browser sandbox escape、kernel脆弱性、
image supply chain、host側container runtimeの権限は残余riskとして残る。

## 仮説・ブレスト `[HYPOTHESIS]`

### 配置案

```text
通常のVS Code／ローカルVenv
  └─ 明示操作: Browser Observe
       └─ 観測専用Dev Container profile
            ├─ Selenium
            ├─ headless Chromium／対応driver
            ├─ 一時user-data-dir
            ├─ allowlistされた要求URLだけ
            └─ 観測receipt
                 ├─ request URL／final URL
                 ├─ HTTP status／headers
                 ├─ DOMまたはartifact hash
                 ├─ browser／driver version
                 ├─ system timestamp＋clock calibration
                 └─ 観測クラス
```

標準Dev Containerへbrowserを常駐bundleせず、`.devcontainer/browser/`に別profileとして置く。通常の
code編集、Skill実行、AI対話はbrowser依存を持たず、必要な検証だけ明示的に結界へ渡す。VS Code側では
標準profileから`include`または選択可能な構成として見せるが、自動起動はしない。

### 結界の初期契約

- hostの認証済みbrowser profile、Cookie、password store、SSH鍵、cloud credentialをmountしない
- Docker socket、host network、不要なport、workspace全体のwrite権限を渡さない
- source checkoutはread-onlyを優先し、成果物は専用出力directoryだけへ返す
- URLは明示入力またはallowlist fixtureに限定し、crawlerや自動巡回にしない
- navigation、script、download、processへtimeoutと容量上限を置く
- downloadは既定無効。一時profile、browser process、download、volume、container、専用Venvまでを
  清掃単位として扱う
- 外部実行前にlocal static fixtureでredirect、DOM mutation、無限loop、cache headers、異常終了を試す
- 観測結果は「安全」「最新」「正しい」に昇格させず、receiptが示す観測範囲だけを返す

対象ページの処理をlocal isolated browser profileへ寄せることは、OpenAI、Anthropicその他SaaSの
model runtimeへ当該ページを実行させることを意味しない。一方、URLや要約をSaaSへ送る別機能があれば
別のdata-flow reviewが必要であり、このprofileだけで保護済みとは扱わない。

手元browser MCPを暫定退避路に使う場合も同じである。利用者host上のbrowserがpageを実行することと、
tool結果がSaaS session contextへ返ることは別の境界としてreceiptへ残す。Atlantisは、XProtect、Safari警告、
任意のAV／EDR、DNS filter、proxy、Fail2ban、privacy relay等を一つの「検疫」へ潰さず、利用者が組んだ
防壁デッキのslotとして宣言する。導入済みという宣言を効力保証へ昇格させない。

Apple系の固有名は今回hostの実験説明にだけ残す。抽象仕様では、利用者が選定したノイマン型計算機のOS、
browser、endpoint／network検疫ツールとして扱う。Atlantisはその選定を邪魔せず、選定と運用は利用者、
製品機能と供給者保証は各providerの仕様・契約範囲とし、どちらの責任も肩代わりしない。

SaaS providerのpolicy、利用者組織のpolicy、法域、World Visaが同じoperationで両立しない場合は、
どれかを普遍的な正義へ昇格せず、そのoperationだけを`scoped_avoid`へ落とす。委託業務範囲外、
provider経路外、ゲームEntity／CompanionのWorld Visa外を別reasonとして保持する。

## 内観メモ `[POEM]`

ブラウザは召喚獣ではなく、入力したWorldのscriptを動かす実行器である。普段の工房へ放し飼いにせず、
観測する時だけ結界へ呼び、観測札を残して結界ごと畳む。結界は無敵バリアではないが、どこまで焦げたら
何を捨てればよいかを先に決める防火区画にはできる。

## 未解決・⊥

- browser containerからのegressをdefault denyにするか、単一host proxy経由にするか
- Chromiumとdriverをimage packageで揃えるか、Selenium Managerを許可するか
- receiptにDOMを保存するかhashだけにするか。第三者著作物・個人情報・秘密値の保持境界
- CDN／service worker／DNS／proxy cacheをどこまで観測し、origin freshnessをどう別証明するか
- robots.txt、利用規約、rate limit、loginをどのauthorityで判定するか
- VS Codeの複数Dev Container構成で、標準profileからどう発見可能にしつつ誤起動を防ぐか
- ARM64、x86_64、Raspberry Pi、Linux、Windows hostでのimage差分と対応表

仕様、神話、UX、権限、privacy、licenseの矛盾を発見した場合は自動debug loopを止める。依存解決、PATH、
browser version整合、fixtureの機械的失敗は、境界契約を変えない範囲で自動修復してよい。
