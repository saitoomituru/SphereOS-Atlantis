## 考察 `[INTERPRETATION]`

### ナラティブ保全

0.1.0は神話名称を単なる飾りへ落とさず、目的関数、約束、棚配置という異なるPositionを保持している。
神話・目的関数の横断工学監査と二重記述憲章にも接続しており、現状は工学へ倒れすぎていない。

ただし実装時に三APIを「branch lint」「fact checker」「formatter」とだけ要約すると、名称が担う認知
チェックサムとポジショントークが消える。Skill本文には物語上のanchorと実務上の問いを両方残す必要がある。

### 人格・神格誤読

0.1.0には`persona gateway`、`architect-class Archangel persona`という歴史語が残る。0.2.0開発環境版で
そのまま前景化すると、Skillを永続人格、天使本体、権威ある命令主体と誤読される可能性がある。

転スラ／ゲームメタファーでは次の分離が有効である。

```text
Skill slot        = 名前付きの監査能力を装備する場所
Skill equip       = 説明・制約・参照先を読み込める状態
Skill invoke      = 対象を指定して一時的に監査すること
Skill output      = Position付きの監査結果。真理・命令・人格発言ではない
Skill holder      = 実行するAI／人間／tool。Skillそのものと同一人格ではない
unmount           = 監査終了。永続自我や常駐人格を残さない
```

「ユニークSkill」という名称は能力の識別子を強くしつつ、本体人格ではないことを伝える。ただし
Skill名だけから権限、結論、宗教的権威を発生させてはならない。

### Position-talk

三Skillは中立を装わず、優先するPositionを明示する。この偏りは機能である。一方、監査対象の作者、
実装者、販売者、事故当事者が同じSkill出力を利用する場合は、Declared Positionとは別に利害位置を記録する。

### 工学と神話の双方向firewall

- Maxwellは未来branchを守るが、未実装を実装済みにはしない
- Urielは証拠境界を守るが、科学・制度だけを全Worldの定規にしない
- Raphaelは共存経路を探すが、危険を無条件trustせず`block`と`unmount`を残す

三Skillを順番に通しても合議制の多数決にしない。出力を別slotのまま保持し、合成結果は新しい
Interpretation OAEとして扱う。
