## 対象・範囲

SphereOS Atlantis `0.250.1` Prompt Engineering Editionから、環境内に存在する異種AI火力を検出・登録・選択・実行・回収できる`0.3xx.n`系列へ移るための要件を洗い出す。

狙いは、MAGIやメインコードAIへ定型処理、待機、キュー管理、provider探索、状態確認を抱え込ませず、トークンコストとレイテンシの痛みをSphere-DOS側の手続きへ降ろすことにある。

このnoteは実装仕様の正本ではない。要件候補、境界、未解決点、Issue分解の起点を保存する。

## 除外範囲

- 自律人格社会やAGI完成の主張

- ローカル専用データの無断クラウド送信

- MAGI Archangel Hostだけを唯一の常駐runtimeとする設計

## 事実・観測 `[FACT]`

1. 現行READMEでは、`0.250.1`をPrompt Engineering Edition、開発環境をSphere-DOS、standalone runtimeを`NOT IMPLEMENTED`としている。
2. Issue #5では、ゲート、キュー、スケジューラー、Idle Cowboy分業を手続き言語へ降ろし、メインAIのトークンコストを抑える案が提示されている。
3. GitHub Issueを共通作業面にすることで、ChatGPT以外のAI環境からも提案を持ち込めることを確認した。
4. 異なるAIが独自のFlavorや命名を持ち込んでも、目的、受入条件、責務、ラベルをIssueへ固定すれば、構造を破局させずに議論を継続できる可能性がある。
5. 現時点で、runtime discovery、provider registry、queue、dispatch、result envelopeを統合したstandalone実装は確認していない。

## 考察 `[INTERPRETATION]`

### 0.250.1から0.3xx.nへの世代境界

`0.250.1`は、Manifest、Prompt、Skill、Gate、Registry、VS Code、異種coding agentを用い、AIが読むべき文脈と越えてはいけない境界を整える世代である。

`0.3xx.n`では、文脈を読むだけでなく、ホスト環境に存在する実行可能なAI火力をSphere-DOSが列挙し、承認済みの火力へジョブを分配し、結果とreceiptを回収する必要がある。

```text
0.250.1 Prompt Engineering Edition
  context / skill / gate / registry中心

0.3xx.n Sphere-DOS runtime generation
  discovery / registry / queue / dispatch / receipt中心
```

この差を越えれば、Atlantisを「複数AI向けプロンプト束」ではなく、最小限の意味で「異種実行主体を扱うマルチエージェントシステム」と説明できる候補になる。

### 常駐デーモンの位置づけ

常駐デーモンはMAGI Archangel Host専用ではなく、ホスト上の利用可能火力を発見し、状態を更新し、ジョブを待ち受けるSphere-DOS側の公共設備として扱う。

MAGIは全ジョブを自分で処理する常駐万能脳ではなく、World、OAE、優先順位、破局境界、最終統合を扱う上位判断系として残す。

定型処理はPython等の手続き実装へ降ろし、推論が必要な箇所だけAI providerへ渡す。

### Flavor耐性

異種AIを同じ口調へ矯正しない。Texas Burger、Cowboy、巫女、ArchangelなどのFlavorはMeaningアセットとして保持できる。

ただし、Flavorが次を上書きしてはならない。

- task目的
- required input
- allowed scope
- output schema
- stop condition
- source provider
- execution receipt

目標は無味無臭化ではなく、異文化エージェントの表現を残したまま、VesselとBridgeを壊さない拘束である。

## 仮説・ブレスト `[HYPOTHESIS]`

### H1. Runtime Discovery

環境内のCLI、ローカル推論器、API adapterを、credential内容へ触れずに検出する。

候補:

- Ollama
- llama.cpp系runner
- MLC系runtime
- Codex CLI
- Claude Code
- Gemini CLI
- GitHub Copilot系入口
- provider-neutral generic CLI adapter
- MCP経由の実行provider

最初から全providerを正式対応しない。`probe`と`adapter`を分離し、Ollamaとgeneric CLI一系統程度からMVPを作る。

### H2. Capability / Position Register

検出したproviderへ、絶対性能ではなく現在デッキ上のポジションを登録する。

```yaml
provider_id: local-ollama-01
provider_type: ollama
state: detected
execution_mode: local
approval_state: unapproved
cost_class: sunk-cost
latency_class: unknown
privacy_class: local
context_class: unknown
strengths: []
weaknesses: []
concurrency: 1
last_probe: null
```

Position-talkは測定済み性能と混同しない。`declared`、`observed`、`benchmarked`を分離する。

### H3. Provider状態遷移

```text
detected
  -> registered
  -> approved
  -> dispatchable
  -> busy / degraded / unavailable
```

自動検出と自動使用を分ける。検出しただけのproviderへ、課金、外部送信、model download、credential取得を行わない。

### H4. Queue / Scheduler

メインコードAIから、待機、再試行、時刻待ち、定型gate、積読処理を剥がす。

最小ジョブ構造:

```yaml
job_id: stable-id
task: string
required_inputs: []
allowed_scope: []
forbidden_scope: []
required_capabilities: []
preferred_cost_class: any
output_schema: result-envelope/0
stop_conditions: []
world_id: null
oae_ref: null
priority: normal
```

Idle Cowboyは、単に暇なAIへ何でも投げる機能ではない。承認済みscopeとcapabilityが一致し、資源上限と停止条件を満たす場合だけdispatchする。

### H5. Result Envelope / Receipt

異種providerの出力を一つの人格へ平均化せず、sourceとFlavorを残して共通封筒へ入れる。

```yaml
job_id: stable-id
provider_id: string
started_at: timestamp
finished_at: timestamp
status: success | partial | failed | refused | timeout
output_ref: string
flavor_preserved: true
claims:
  - scope: string
    strength: fact | interpretation | hypothesis | poem
warnings: []
resource_receipt:
  token_usage: unknown
  wall_time_ms: unknown
  local_compute: unknown
```

### H6. Main AI Token-Pain Boundary

メインAIは次に集中する。

- 問いの分解
- World / OAE / Agency境界
- provider選択方針
- 破局監視
- 出力間の対立整理
- 最終統合

メインAIへ渡さない候補:

- provider存在確認
- バージョン照会
- queue polling
- retry待機
- cron
- 単純なSchema validation
- log rotation
- 定型的なreceipt生成

### H7. CLI最小入口

```text
atlantis agents scan
atlantis agents list
atlantis agents approve <provider-id>
atlantis task submit <task-file>
atlantis task status <job-id>
atlantis task result <job-id>
```

CLI名とSchemaは未確定。既存`atlantis_cli`責務と衝突する場合はsubcommandまたは別moduleへ分離する。

### H8. Scale-firstの意味

0.3xx.n初期段階で先に載せる「scale」は、巨大クラスタや無制限並列ではない。

- provider adapterを追加できる
- provider数が増えても同じqueue契約を使える
- local / cloudを同じsourceへ潰さない
- concurrency上限をproviderごとに持てる
- failureを一つのAIの人格障害として扱わず、worker状態として切り離せる
- メインAIのcontext windowをworker数に比例して肥大化させない

この構造が成立してから、優先度、負荷分散、benchmark、quota、transaction、World別routingを拡張する。

## 安全・権限境界

- 実行ファイルの存在確認は許可候補だが、任意command実行は別承認とする。
- `--version`等もproviderごとの安全probe allowlistを持つ。
- shell history、environment variable、config fileからsecretを収集しない。
- ログへtoken、credential、個人情報、enterprise pathを出さない。
- 外部送信を伴うproviderは、初回およびscope変更時に承認を要求する。
- providerが見つからない場合、存在を捏造せず`BOTTOM`または`unavailable`を返す。
- 高額課金、model download、network常駐、daemon installは自動実行しない。
- host OSのサービス登録は、read-only MVPと分離する。
