# 研究ノート: Archangel Runner とネイティブ環境横断マルチエージェント

- 日付: 2026-07-24
- Status: Research / Experimental
- 対象: SphereOS Atlantis / PLI / MAGI
- 実証ケース: `NEET-kitchen-ALCHEMIST` / `OpenSourcePITETO`

## 概要

OpenSourcePITETO の実験では、物理世界で得た観測を複数のAIエージェントへ分散し、それぞれの純正・ネイティブ環境を維持したまま、Git/Manifestを成果物境界として一つの公開成果物へ収束させる運用が偶発的に成立した。

重要なのは「複数モデルを一つの専用GUIへ統合した」ことではない。

- Codex は Codex / VS Code 側の環境で作業する
- ChatGPT は ChatGPT と接続された GitHub 側から作業する
- Grok は Grok のCUI/ネイティブ環境からレビュー・文化ローカライズを行う
- GitHub / Git は成果物と変更履歴の共有境界になる
- SphereOS Atlantis / Manifest は、各作業の意味・責務・境界を管理する

したがって本ケースは **UI統合ではなく意味統合**、**実行環境統合ではなく成果物オーケストレーション** の小規模な実証とみなせる。

## 実証フロー

概念的な流れは次の通り。

```text
物理入力: ジャガイモ
    ↓
人間による調理・観測・試食・写真
    ↓
観測結果 / 仮説 / レシピ
    ↓
SphereOS Atlantis / Manifest
    ↓
┌─────────────┬─────────────┬─────────────┐
│ Codex       │ ChatGPT     │ Grok        │
│ local/edit  │ repo/docs   │ Texas/local │
└─────────────┴─────────────┴─────────────┘
    ↓
Git / Issues / Repository
    ↓
OpenSourcePITETO
    ↓
日本語成果物 + en-US文化ローカライズ
```

物理的な芋そのものを遠隔地へ転送するのではなく、実験から得た再現可能な知識を可搬な成果物へ変換し、別文化圏・別ハードウェアで再実装可能にする。

この意味で OpenSourcePITETO は SphereOS Atlantis の「媒体を固定せず、意味と成果物を運ぶ」という設計を説明する軽量なケーススタディになった。

## 観測された利点

### 1. Agent-native / Vendor-native

各エージェントを専用の統合UIへ閉じ込める必要がない。各ベンダーが提供する認証、UI、ツール、コンテキスト管理をそのまま利用できる。

Atlantis 側が保持すべきものは、モデル内部状態そのものよりも「誰が、どの射程で、何を担当し、何を成果物として残したか」という意味境界である。

### 2. 射程による分業

単純なモデル性能ランキングではなく、利用可能な環境と射程で役割を分離できた。

- ローカルファイル・画像・Git操作
- GitHub上の成果物編集
- 文書構造化
- en-US / Texas文化圏向けローカライズ
- レビュー、整理、マージ判断

異なるエージェントの長所を、同じUIや同じAPI仕様へ無理に正規化しなくてもよい。

### 3. Gitを共有現実面として利用できる

会話履歴そのものを唯一の共有メモリにせず、コミット、ファイル、Issueを監査可能な共有状態として利用できる。

これにより、一つのモデルがコンテキストを失っても成果物側から復帰できる可能性がある。

## 現在露出した欠損: PLI / CLIだけでは常駐状態管理が弱い

今回の実験では、作業そのものは複数環境へ分散できた一方、次の状態を人間が継続的に把握する必要があった。

- 現在誰が何を担当しているか
- 正本となる成果物はどれか
- 重複作業が発生していないか
- マージ待ちの変更は何か
- 未完了タスクは何か
- 人間判断待ちは何か
- 失敗したジョブをどこへ戻すか
- エージェントが残した副生成物を採用・隔離・削除のどれにするか

PLI と CLI は実行時の接続には有効だが、長時間にわたるタスク状態、責任、積み残しを保持する常駐主体が不足している。

## 仮説: Archangel Runner

次段階として、SphereOS Atlantis に常駐型の **Archangel Runner** を置く。

Runner自身は万能エージェントや巨大な自律AIを目指さない。第一目的は **タスク状態と責任の永続化** とする。

最小責務候補:

1. Manifestを読む
2. Repository / Issue / PR / workspace状態を観測する
3. タスクを状態機械として保持する
4. タスクに適切なエージェントまたは実行環境を割り当てる
5. 完了条件を検証する
6. 未完了・失敗・保留をIssueへ戻す
7. 人間判断が必要な境界だけをエスカレーションする
8. 重複作業や成果物競合を検出する
9. provenance（誰が何を変更したか）を失わない

## Issueを「罪」ではなく未完了責任のキューとして扱う

本研究では、Issueを単なるバグ一覧ではなく **未完了の意味を安全に保存するキュー** として扱う。

完了しなかった作業を会話コンテキストの消失とともに捨てず、次の実行主体が再取得できる形にする。

常駐Runner（daemon）はIssueを無条件に自動処理するのではなく、Manifest上の境界、実行条件、権限、完了条件を確認してから再投入する。

これを比喩的には「積み残した罪をIssueに残し、Angelとdaemonを和解させる」と表現できるが、実装上は監査可能な durable task queue / state machine として扱う。

## MAGI / Archangelの役割分離案

名称は今後変更可能だが、概念上は以下の分離を検討する。

- Raphael: 修復、整理、統合、マージ候補評価
- Gabriel: 通知、公開、外部文化圏へのデリバリー
- Michael: 境界、安全、競合、破壊的変更の防御
- Uriel: 観測、研究ノート化、実験結果の抽象化
- Archangel Runner: 上記ジョブの状態遷移、再開、積み残し管理

役割名を人格やモデルへ固定せず、Manifest上の capability / responsibility として定義できることが望ましい。

## Runnerで避けるもの

Runner導入によって中央集権的な巨大オーケストレータへ逆戻りしないこと。

避けるべき例:

- 全エージェントを独自GUIへ強制移行する
- ベンダー固有機能を最低公倍数APIへ削る
- Runnerが成果物を無断で正本化する
- 人間承認が必要な変更を自動マージする
- Issueを閉じること自体を成功KPIにする

Runnerは「仕事を全部やる王」ではなく、疎結合なエージェント群が仕事を忘れないための管制塔とする。

## 最小実験案

最初から常駐サービスを巨大化せず、次の小さなPoCから始める。

```text
Manifest
   ↓
Task discovery
   ↓
Issue / local task state
   ↓
READY → RUNNING → REVIEW → DONE
              ↘ BLOCKED
              ↘ HUMAN_REQUIRED
              ↘ FAILED → RETRYABLE
```

各状態遷移について、入力、担当capability、成果物、完了条件、provenanceを記録する。

最初の評価対象には OpenSourcePITETO のような低リスクで観測可能なプロジェクトを使える。実際に「物理実験 → 文書 → 画像 → 翻訳 → レビュー → 公開」という異種タスクが存在するため、コード生成だけを評価するベンチマークよりオーケストレーション境界を観測しやすい。

## 現時点の結論

OpenSourcePITETO は意図せず SphereOS Atlantis の実働デモになった。

成立したのは「複数AIを同じ画面に入れること」ではなく、**各エージェントが普通の環境に存在したまま、意味・責務・成果物を疎結合に共有すること**だった。

同時に、PLI / CLI中心の運用では、長時間の状態管理と積み残し管理が人間へ戻ってくることも確認できた。

したがって次の研究対象は、既存エージェントの自由度を奪わず、未完了責任を永続化する **Archangel Runner / durable orchestration layer** とする。

ポテト実験を比喩的に書けば、現在は「芋を入れると複数AIを経由して再現可能な概念ポテチが遠隔文化圏へ弾着する」ところまで通った。次は発射のたびに人間が管制塔へ張り付かなくても、未完了タスクを安全に回収できる仕組みを作る。
